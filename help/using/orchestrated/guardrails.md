# 가드레일 및 제한 사항 {#guardrails}

오케스트레이션된 캠페인을 사용할 때는 다음과 같은 보호 기능 및 제한 사항이 있습니다.

## 데이터 흐름 제한

### 데이터 디자인 및 스토리지

* **최대 테이블** - 관계형 데이터 저장소는 최대 200개의 테이블(스키마)을 지원합니다.

* **스키마 크기** - 오케스트레이션된 캠페인의 경우 개별 스키마의 총 크기는 100GB를 초과할 수 없습니다.

* **일별 업데이트 볼륨** - 성능과 안정성을 유지하려면 스키마의 일별 업데이트를 총 레코드 수의 20% 미만으로 제한해야 합니다.

* **관계형 데이터 모델** - 관계형 데이터는 수집, 데이터 모델링 및 세그먼테이션 사용 사례에 지원되는 기본 모델입니다.

* **ID 필드** - 타깃팅에 사용되는 스키마에는 정의된 ID 네임스페이스에 매핑된 `String` 유형의 ID 필드가 하나 이상 있어야 합니다.

* **스키마당 특성** - 관리 용이성과 성능을 유지하려면 스키마당 평균 특성 수가 50열을 초과할 수 없습니다.

* **프로필 사용** - Adobe Experience Platform 프로필에 대해 관계형 스키마를 사용할 수 없습니다. Adobe Experience Platform 프로필에는 표준 XDM 스키마만 지원됩니다. 오케스트레이션된 캠페인 또는 작업 캠페인에 대해 관계형 스키마를 활성화할 수 있습니다. [자세히 알아보기](https://experienceleague.adobe.com/ko/docs/experience-platform/catalog/datasets/user-guide#enable-profile)

### 데이터 수집 {#data-ingestion}

* **프로필 및 관계 수집** - 프로필 + 관계 데이터 수집이 필요합니다.

* **데이터 캡처 원본 변경** - 모든 수집은 데이터 캡처 원본 변경을 통해 수행되어야 합니다.

   * **파일 기반 원본** - `_change_request_type` 필드는 필수입니다. 지원되는 값은 `u`(업데이트) 또는 `d`(삭제)입니다. 이 값은 대문자 `U` 및 `D`이(가) 아닌 소문자 `u` 및 `d`(으)로 입력해야 합니다.

   * **클라우드 기반 원본** - 테이블 로깅을 사용하도록 설정해야 합니다.

* **레코드 완료만** - 부분 레코드 업데이트는 허용되지 않습니다. 각 행을 전체 레코드로 제공해야 합니다.

* **일괄 처리 수집 빈도** - Campaign Orchestration의 일괄 처리 수집은 15분마다 한 번으로 제한됩니다.

* **수집 대기 시간** - 관계형 저장소의 수집 대기 시간은 일반적으로 다음에 따라 15분에서 2시간 사이입니다.

   * 데이터 볼륨

   * 시스템 동시 실행

   * 작업 유형(예: 삽입이 업데이트보다 빠름)

* **데이터 집합 관계에 대한 데이터 흐름** - 데이터 집합 관계에 대한 데이터 흐름은 1-1입니다. 하나의 소스만 주어진 시간에 하나의 데이터 세트를 제공할 수 있습니다. 소스를 전환하려면 기존 데이터 흐름을 삭제하고 새 소스로 새 데이터 흐름을 만듭니다.

### 데이터 모델링

* **버전 설명자** - 팩트 테이블을 포함한 모든 스키마에는 올바른 버전 제어 및 추적을 위해 버전 설명자가 포함되어야 합니다.

* **기본 키** - 데이터 무결성 및 다운스트림 작업을 지원하려면 각 테이블에 정의된 기본 키가 있어야 합니다.

* **영구 테이블 이름** - 데이터 집합을 만드는 동안 할당된 `table_name`은(는) 영구적이며 세분화 및 개인화 기능 전체에서 사용됩니다.

* **필드 그룹** - 필드 그룹은 현재 데이터 모델링 프레임워크에서 지원되지 않습니다.

* **복합 기본 키** - 현재 파일 업로드 흐름이 있는 복합 기본 키에 대한 지원을 사용할 수 없습니다.

## 활동 제한 사항 {#activities-limitations}

* **채널 활동 제한** - 오케스트레이션된 캠페인은 최대 10개의 채널 활동(전자 메일, SMS, 푸시 또는 DM)을 지원합니다. 채널 활동만 이 제한에 포함됩니다. 타깃팅 및 흐름 제어 활동은 계산되지 않습니다(예: 대상 작성, 대기, 분할, 데이터 보강, 조정, 포크, 종료 또는 테스트).

  저장 또는 게시 시 한도를 초과하면 작업이 실패합니다. 이 한도를 유지하려면 채널 활동 수를 줄이거나 오케스트레이션된 여러 캠페인에 메시지 게재를 분할합니다.

* **캔버스 활동 제한** - 오케스트레이션된 캠페인 캔버스의 활동 수는 500개로 제한됩니다. 이 제한은 캔버스의 모든 활동 유형에 적용됩니다. 게시 시 적용되는 채널 활동 제한과 별개입니다. 유지 관리 및 성능을 위해 워크플로우를 100개 미만으로 유지하십시오.

* **스칼라 특성만** - 스칼라 특성만 대상 정의에서 지원됩니다. 맵과 배열은 허용되지 않습니다.

* **세그먼테이션의 관계형 데이터** - 세그먼테이션 활동은 주로 관계형 데이터에 의존합니다. 프로필 데이터를 포함할 수 있지만 큰 프로필 데이터 세트를 사용하면 성능에 영향을 줄 수 있습니다.

* **프로필 특성 제한** - 시스템 효율성을 유지하기 위해 일괄 처리와 스트리밍 대상 모두에서 사용할 수 있는 프로필 특성 수에 제한이 적용됩니다.

* **열거형** - 열거형이 완전히 지원됩니다.

* **대상자 읽기가 캐시되지 않음** - 대상자 읽기가 캐시되지 않습니다. 각 캠페인 실행은 기본 데이터에서 전체 대상자 평가를 트리거합니다.

* **대상 최적화** - 성능을 보장하기 위해 크거나 복잡한 대상 정의로 작업할 때는 최적화를 사용하는 것이 좋습니다.

* **저장된 대상은 정적입니다** - 저장된 대상 활동은 정적입니다. 이 활동은 캠페인 실행 시 사용 가능한 데이터를 반영합니다.

* **저장된 대상에 추가하지 않음** - 저장된 대상 활동에 추가할 수 없습니다. 모든 수정 사항을 적용하려면 대상을 완전히 덮어써야 합니다.

## 채널 제한 사항

오케스트레이션된 캠페인에서는 SMS, 푸시, 이메일 및 DM 채널만 지원됩니다.
