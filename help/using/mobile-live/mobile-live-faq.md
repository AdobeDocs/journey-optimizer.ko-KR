## 일반 질문

+++라이브 활동과 푸시 알림의 차이점은 무엇입니까?

라이브 활동은 사용자가 디바이스를 잠금 해제할 필요 없이 잠금 화면 및 Dynamic Island에서 지속적인 실시간 업데이트를 제공합니다. 푸시 알림은 무시되면 사라지는 임시 경고입니다. 라이브 활동은 계속 표시되며 명시적으로 종료될 때까지 여러 번 업데이트할 수 있습니다.

+++

+++한 번에 얼마나 많은 라이브 활동을 활성화할 수 있습니까?

iOS 앱은 동일한 `ActivityAttributes` 유형을 사용하는 여러 라이브 활동을 포함하여 여러 라이브 활동을 동시에 실행할 수 있습니다.

개발자는 특정 속성 유형의 라이브 활동 수에 엄격한 제한을 두지 않습니다. 예를 들어 지속적인 게재 또는 라이드에 대해 앱 논리에 필요한 만큼 시작할 수 있습니다. 그러나 iOS에서는 활성 상태이거나 한 번에 표시되는 라이브 활동 수에 대한 시스템 수준 제한을 적용합니다.

실제로:

* iOS은 일반적으로 앱당 최대 5개의 동시 라이브 활동을 지원합니다.

* 이 수를 초과하면 일부 활동 표시를 중단하거나 이전 활동을 종료하여 리소스를 절약할 수 있습니다.

* 각 라이브 활동에는 개별적으로 업데이트하거나 종료할 수 있는 고유한 `Activity.id`이(가) 있습니다.

+++

+++라이브 활동 업데이트를 수신하려면 앱을 열어 두어야 합니까?

아니요. 이 기능의 주요 이점 중 하나인 앱을 완전히 닫아도 라이브 활동을 원격으로 시작, 업데이트 및 종료할 수 있습니다.

+++

+++라이브 활동을 지원하는 iOS 버전은 무엇입니까?

* iOS 16.1+: 기본 라이브 활동 지원
* iOS 17.2+: Push-to-Start 기능(앱을 열지 않고 원격으로 시작)
* iOS 18+: 대상 기반 라이브 활동에 대한 브로드캐스트 채널 지원
+++

+++라이브 활동은 얼마나 오랫동안 활성 상태를 유지할 수 있습니까?

Apple은 라이브 활동을 **활성 업데이트 시간**(으)로 제한합니다. 그 후에도 시스템은 활동을 자동으로 종료하지만 제거하기 전에 최대 **12시간** 동안 정적 상태로 계속 볼 수 있습니다. `dismissalDate`을(를) 설정하거나 앱에서 `activity.end()`을(를) 명시적으로 호출하여 라이브 활동을 더 빨리 종료할 수도 있습니다.

+++

### 개발자 질문

+++라이브 활동에 대해 별도의 위젯 확장을 만들어야 합니까?

예. 라이브 활동은 WidgetKit을 통해 표시되므로 Xcode 프로젝트에서 위젯 확장을 만들고 `ActivityConfiguration`을(를) 구현해야 합니다.
[위젯 구성에 대해 자세히 알아보기](mobile-live-configuration-sdk.md)

+++

+++로컬 및 원격 라이브 활동 모두에 동일한 `LiveActivityAttributes` 클래스를 사용할 수 있습니까?

예. 동일한 속성 클래스는 로컬에서 시작한 라이브 활동과 원격으로 시작한(push-to-start) 라이브 활동 모두에 대해 작동합니다. `Messaging.registerLiveActivity()`에 등록해야 합니다.

+++

+++존재하지 않는 라이브 활동에 대한 업데이트를 보내면 어떻게 됩니까?

존재하지 않는 `liveActivityID` 또는 `channelID`에 대한 업데이트 또는 종료 이벤트를 보내는 경우 장치에서 요청이 자동으로 실패합니다. 각 사용자에 대해 활성화된 라이브 활동을 항상 추적하는지 확인하십시오.

+++

+++iOS 시뮬레이터에서 라이브 활동을 테스트할 수 있습니까?

예. iOS 시뮬레이터에서 로컬에서 시작한 활동과 원격으로 시작한 라이브 활동을 테스트할 수 있습니다.

* **로컬**: 여기에는 **ActivityKit API**&#x200B;를 사용하여 앱에서 직접 라이브 활동을 만들고, 업데이트하고, 종료하는 작업이 포함됩니다.

* **원격**: 라이브 활동 기능을 원격으로 테스트하려면 메시징 SDK을 앱에 통합하고 제공된 실행 API를 사용하여 테스트 장치 또는 iOS 시뮬레이터로 원격 시작, 업데이트 및 종료를 보냅니다. 현재 Adobe SDK 통합을 통해 푸시 알림을 테스트할 수 있는 방법과 유사합니다.

+++

+++앱이 백그라운드에 있을 때 업데이트를 처리하려면 어떻게 해야 합니까?

SDK이 이 작업을 자동으로 처리합니다. 등록되면 앱이 종료된 경우에도 라이브 활동이 업데이트를 받습니다. 추가적인 배경 모드는 필요하지 않습니다.
+++

+++`liveActivityID`과(와) `channelID`의 차이점은 무엇입니까?

* `liveActivityID`: 특정 사용자를 대상으로 하는 개별(단일) 라이브 활동에 사용됩니다. 각 ID는 고유한 라이브 활동 인스턴스를 나타냅니다.
* `channelID`: 대상자에게 전송된 브로드캐스트 라이브 활동에 사용됩니다. 대상의 모든 사용자는 동일한 채널에서 동일한 업데이트를 받습니다.
+++

+++잠금 화면과 별도로 동적 아일랜드 모양을 사용자 정의할 수 있습니까?

예. `ActivityConfiguration`은(는) 잠금 화면 콘텐츠와 동적 섬 콘텐츠(확장, 축소 및 최소 상태)에 대해 각각 독립적으로 닫혀 있습니다.
+++

+++푸시 토큰을 수동으로 저장해야 합니까?

아니요. `Messaging.registerLiveActivity()`에 라이브 활동 유형을 등록하면 SDK에서 자동으로 푸시 토큰을 수집하고 관리합니다.
+++

### 마케터 질문

+++브로드캐스트 캠페인의 각 사용자에 대한 라이브 활동 컨텐츠를 개인화할 수 있습니까?

브로드캐스트 캠페인은 대상의 모든 사용자에게 동일한 콘텐츠를 전송합니다. 개인화된 콘텐츠의 경우 개별 사용자를 타겟팅하는 단일(트랜잭션) 캠페인을 사용하십시오.
+++

+++내 라이브 활동이 성공적으로 전달되었는지 어떻게 알 수 있습니까?

Adobe Journey Optimizer에서 [캠페인 분석 모니터링](../reports/campaign-global-report-cja-activity.md). 게재 비율, 실패 및 참여 지표를 추적할 수 있습니다. 앱에서 사용자 지정 분석 이벤트를 구현하는 방법도 고려해 보십시오.
+++

+++라이브 활동을 미리 예약할 수 있습니까?

API 호출은 라이브 활동을 즉시 트리거합니다. 그러나 백엔드 시스템을 통해 API 호출을 예약하거나 Journey Optimizer의 오케스트레이션 기능을 사용하여 적절한 시간을 할당할 수 있습니다.
+++

+++이미 존재하는 라이브 활동에 대해 &quot;시작&quot; 이벤트를 보내면 어떻게 됩니까?

Adobe의 실행 API를 통해 라이브 활동을 원격으로 시작할 때:

* 요청에 `x-request-id` 헤더를 포함할 수 있습니다. 각 `liveActivityID`과(와) 해당 `x-request-id` 사이에 일대일 관계가 있는 것이 좋습니다. 이렇게 하면 동일한 `x-request-id` 및 `liveActivityID` 조합으로 여러 요청을 수행하는 경우 장치에서 하나의 라이브 활동만 시작되고 중복 요청은 무시됩니다.

* `x-request-id` 헤더가 생략되면 각 요청이 독립적으로 처리되어 동일한 `liveActivityID`을(를) 사용하여 여러 라이브 활동이 만들어질 수 있습니다. 이러한 경우 향후 업데이트가 실패하거나 활성 인스턴스 중 하나에만 적용될 수 있습니다.

* `x-request-id` 값은 별도의 API 요청에서 다른 `liveActivityIDs`에서 재사용해서는 안 됩니다.

+++

+++A/B에서 다른 라이브 활동 경험을 테스트할 수 있습니까?

예. 다양한 콘텐츠 구조를 사용하는 여러 캠페인을 만들고 Adobe Journey Optimizer의 실험 기능을 사용하여 성과가 더 좋은 제품을 테스트합니다. 앱이 모든 콘텐츠 상태 변형을 지원하는지 확인합니다.

+++

+++라이브 활동은 얼마나 자주 업데이트해야 합니까?

너무 자주 업데이트하면 배터리가 소진되고 사용자 경험 품질이 저하될 수 있으므로 의미 있는 정보가 변경되는 경우에만 업데이트합니다. 게재 추적과 같은 실시간 시나리오의 경우 일반적으로 30~60초마다 사용할 수 있습니다. 스포츠 점수와 같이 더 느리게 변경되는 콘텐츠의 경우 중요한 이벤트에서만 업데이트합니다.

+++

+++라이브 활동이 활성화되어 있는지 여부에 따라 사용자를 타깃팅할 수 있습니까?

개발 팀과 함께 이 환경 설정을 추적하여 사용자 속성으로 Adobe Experience Platform에 전달한 다음 해당 속성을 기준으로 세그먼트화해야 합니다.

+++

### API 질문

+++`timestamp`과(와) `dismissal-date`의 차이점은 무엇입니까?

* `timestamp`: 이벤트가 발생하는 현재 epoch 시간으로, 모든 이벤트에 필요합니다.
* `dismissal-date`: 라이브 활동이 자동으로 해제되어야 하는 향후 epoch 시간으로, &quot;종료&quot; 이벤트에만 필요합니다.

+++

+++모든 업데이트 호출에서 모든 `attributes` 필드를 보내야 합니까?

예, `LiveActivityAttribute` 클래스를 기반으로 합니다.

* `liveActivityData`을(를) 포함한 특성 개체의 모든 필드는 시작, 업데이트 또는 종료에 대한 모든 호출에 포함되어야 합니다.
* `content-state` 필드만 실행 중인 라이브 활동에서 실제로 동적으로 변경되는 내용을 나타냅니다.
* 경고 개체도 포함합니다. 이 개체는 푸시가 자동 배경이 아닌 사용자가 볼 수 있는 알림으로 처리됩니다. &#39;시작&#39; 사례에만 필요하며 그 외의 경우에는 선택 사항입니다.

+++

+++에포크 타임스탬프는 어떤 형식으로 표시되어야 합니까?

Unix epoch 시간은 밀리초가 아닌 **초**&#x200B;로 사용합니다. 예: `1759937682`

+++

+++여러 API 호출에 동일한 `requestId`을(를) 사용할 수 있습니까?

아니요. 동일성과 적절한 추적을 보장하려면 각 API 요청에 고유한 `requestId`이(가) 있어야 합니다. UUID 또는 유사한 고유 식별자를 사용합니다.

+++

+++Headless API에 필요한 인증은 무엇입니까?

OAuth 토큰 및 API 키를 포함한 인증 요구 사항은 [API 트리거된 캠페인 설명서](https://developer.adobe.com/journey-optimizer-apis/references/messaging/)를 참조하십시오.

+++

+++API 호출이 실패하면 어떻게 됩니까?

지수 백오프를 사용하여 재시도 논리를 구현합니다. API 응답에서 오류 코드 및 메시지를 확인하여 문제를 진단합니다. 일반적인 실패에는 잘못된 캠페인 ID, 잘못된 페이로드 또는 인증 문제가 포함됩니다.

+++

+++자체 백엔드 서버에서 라이브 활동 업데이트를 보낼 수 있습니까?

네, 그것은 의도된 행동입니다. 백엔드는 비즈니스 로직에 라이브 활동 이벤트가 필요한 경우 Adobe Journey Optimizer Headless API를 호출하여 라이브 활동 이벤트를 트리거합니다.

+++

+++시작, 업데이트 및 종료 이벤트에 다른 캠페인이 필요합니까?

아니요. 동일한 캠페인을 사용하고 페이로드에서 `event` 필드를 변경할 수 있습니다. 그러나 일부 조직에서는 더 나은 분석 추적을 위해 별도의 캠페인을 선호합니다.

+++

### 문제 해결

+++내 라이브 활동이 시작되지만 업데이트되지 않습니다. 무엇이 문제일 수 있습니까?

일반적인 원인:

* 시작 호출과 업데이트 호출 사이에 `liveActivityID` 또는 `channelID`이(가) 일치하지 않습니다.
* `content-state` 필드가 `ContentState` 구조체와 일치하지 않습니다.
* 라이브 활동이 이미 종료되었습니다.
* 장치의 네트워크 연결 문제입니다.

+++

+++`attributes-type` 필드를 인식할 수 없습니다. 무엇을 확인해야 합니까?

* 클래스 이름이 Swift 구조 이름과 **exactly**(대/소문자 구분)와 일치하는지 확인하십시오.
* 구조가 올바르게 정의 및 등록되었는지 확인
* JSON 페이로드에서 오타 확인
* 설치된 앱 버전에 라이브 활동 구현이 있는지 확인

+++

+++사용자에게는 라이브 활동 업데이트만 표시되고 경고 알림은 표시되지 않습니다. 알려진 문제입니까?

아니요. `alert` 필드는 선택 사항이며, [방해 금지] 모드와 같은 특정 조건에서 iOS에 의해 금지될 수 있습니다. 라이브 활동은 자동으로 업데이트될 수 있으며, 이는 종종 의도된 동작입니다. 경고 필드는 원격 시작을 전송하는 데 필수입니다. 그렇지 않으면 apple은 이를 자동 백그라운드 알림으로 처리합니다.

+++

+++사용자의 모든 라이브 활동을 삭제하거나 지울 수 있습니까?

각 활성 라이브 활동에 대해 &quot;종료&quot; 이벤트를 보내야 합니다. 시스템에서 각 사용자에 대해 활성화된 라이브 활동을 추적하여 적절하게 정리할 수 있습니다.

+++업데이트를 보냈는데도 내 위젯에 &quot;데이터 없음&quot;이 표시됩니다. 무엇이 문제일 수 있습니까?

* 위젯 구현이 `context.state` 및 `context.attributes`에 올바르게 액세스하는지 확인하십시오.
* 위젯 인터페이스에서 기본값 또는 오류 상태가 처리되는지 확인하십시오.
* `LiveActivityAssuranceDebuggable` 프로토콜을 사용하여 스키마를 디버그합니다.
* Adobe Assurance으로 테스트하여 데이터가 수신되는지 확인합니다.
